# RULES

- [2026-04-05 05:53:47] Рабочий язык проекта и документации в `docs/` — русский.
- [2026-04-05 06:02:47] Вся документация, комментарии, описания изменений и рабочее взаимодействие ведутся на русском языке; английский допускается только для технических терминов, имён файлов, идентификаторов, протоколов и названий переменных.
- [2026-04-05 05:53:47] После каждого логического этапа разработки документация в `docs/` должна обновляться.
- [2026-04-05 05:53:47] Все записи в `docs/` фиксируются с таймстемпом в формате `[ГГГГ-ММ-ДД ЧЧ:ММ:СС]`.
- [2026-04-05 05:53:47] Перед внедрением изменений сначала проводится анализ текущего состояния, затем формируется план, затем вносятся правки.
- [2026-04-05 05:53:47] Коммиты должны быть атомарными и по формату `Conventional Commits`.
- [2026-04-05 07:01:41] Канонический GitHub-репозиторий проекта — `origin = https://github.com/RM-COM/AT-VPN-System.git`, а исходный репозиторий для синхронизации изменений — `upstream = https://github.com/mozaroc/x-ui-pro.git`.
- [2026-04-05 07:01:41] Базовая ветка форка — `main`; все новые этапы доработок рекомендуется вести через отдельные feature-ветки с последующим атомарным merge.
- [2026-04-08 04:30:47] Перед удалением, архивированием или массовой чисткой веток их роли и наличие уникальных коммитов должны быть явно зафиксированы в `docs/BRANCH_MATRIX.md`.
- [2026-04-08 04:30:47] Перед физическим архивированием или удалением документов их роль должна быть явно зафиксирована в `docs/DOCS_MATRIX.md`, а канонический слой документации должен быть обновлён заранее.
- [2026-04-05 07:06:52] Канонический рабочий цикл проекта: сначала изменения вносятся и проверяются на Windows в локальном репозитории, затем отправляются в GitHub (`origin`), и только после этого выгружаются на Linux staging для запуска и тестов.
- [2026-04-05 07:23:54] После каждого завершённого логического этапа, в котором были фиксы, debug-изменения, доработки или правки документации, изменения должны быть зафиксированы отдельным commit'ом и отправлены push'ем в `origin` до перехода к следующему этапу.
- [2026-04-11 10:28:01] При проектировании transport-профилей, fallback и migration-сценариев нужно сохранять стабильный клиентский контракт: штатное обновление подписки допустимо, но полная переимпортировка подписки и ручная перенастройка клиентов считаются product-breaking изменением, если это не было отдельно запланировано и явно задокументировано.
- [2026-04-05 07:06:52] Перед каждым новым полноценным staging-прогоном сервер должен приводиться в предсказуемое чистое состояние: через поддерживаемый `uninstall`/reset-сценарий или документированный эквивалентный workflow очистки.
- [2026-04-05 07:06:52] Сценарии полного удаления, резервного копирования и восстановления считаются обязательной частью форка; `uninstall`, `backup` и `restore` нельзя терять или ломать при рефакторинге.
- [2026-04-05 05:53:47] Чувствительные данные не хардкодятся; для переменных окружения поддерживается `.env.example`.
- [2026-04-10 20:16:31] В GitHub (ветки, commit-история, `docs/`, `README`, HTML-отчёты) разрешён только project-safe контент: техническая информация о проекте, изменениях, тестах и плане без личного контекста.
- [2026-04-10 20:16:31] Личный контекст, приватные обсуждения и персональные заметки хранятся только локально вне публично-распространяемого контента репозитория.
- [2026-04-10 20:16:31] Если личный контент случайно попал в проектные документы, перед следующим push он должен быть заменён на нейтральный технический текст.
- [2026-04-10 20:16:31] Branch HTML-страницы и branch-specific README/документы в GitHub должны содержать только project-safe описания функционала, статуса и плана ветки без персональных деталей.
- [2026-04-07 04:00:36] Для крупных архитектурных разборов, аудитов и исследовательских этапов нужно использовать параллельных агентов там, где это ускоряет работу без потери качества; итоговое решение всё равно сводится в единый основной design-doc внутри `docs/`.
- [2026-04-05 05:53:47] При команде пользователя `Добавь правило: ...` новая запись должна быть немедленно добавлена в этот файл с актуальным временем.
