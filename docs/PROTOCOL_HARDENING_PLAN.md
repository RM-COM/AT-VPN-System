# PROTOCOL_HARDENING_PLAN

## Назначение

- [2026-04-10 22:48:10] Этот документ фиксирует технологический план усиления протоколов форка: снижение DPI-сигнатур, устойчивость к `RST`/drop-деградации, защита админского контура и порядок дальнейшей реализации.
- [2026-04-10 22:48:10] Документ не заменяет `XRAY_DPI_PLAN.md` и `ANTI_DPI_MATRIX.md`, а уточняет, какие именно технологии и инженерные подходы будут использоваться внутри активного `Block C1`.
- [2026-04-10 22:48:10] Публикуемый контент намеренно project-safe: без личного контекста, секретов, доменных паролей, приватных логов и данных пользователей.

## Ключевой вердикт

- [2026-04-10 22:48:10] Главная защита от `RST`-разрывов — не попытка "победить RST" после детекта, а снижение вероятности самого детекта.
- [2026-04-10 22:48:10] `RST`, packet drop и timeout рассматриваются как симптомы классификации или сетевого воздействия, а не как единственная корневая причина.
- [2026-04-10 22:48:10] Целевая модель проекта: `невидимость до детекта + устойчивость после детекта + быстрый fallback + сохранение скорости`.
- [2026-04-10 22:48:10] Один "магический" протокол не считается надёжной стратегией. Правильная форма форка — controlled multi-transport платформа с понятными профилями и проверками.

## Проверенные внешние источники и выводы

- [2026-04-10 22:48:10] Официальная документация Xray по `WebSocket` прямо указывает, что `WebSocket` имеет заметную traffic characteristic через `ALPN is http/1.1` и рекомендует переходить на `XHTTP`: https://xtls.github.io/en/config/transports/websocket.html
- [2026-04-10 22:48:10] Официальная страница Xray `XHTTP: Beyond REALITY` ведёт на актуальное обсуждение Xray-core по `XHTTP`, где описаны `packet-up`, `stream-up`, header padding, `XMUX`, H2/H3 и split up/down model: https://xtls.github.io/en/config/transports/xhttp.html и https://github.com/XTLS/Xray-core/discussions/4113
- [2026-04-10 22:48:10] Документация Xray по `fallbacks` подтверждает важное ограничение: `fallbacks` применимы к `TCP+TLS`, поэтому для `REALITY` нельзя слепо переносить TLS fallback-паттерны; в нашем `stealth`-профиле source of truth остаётся профильный `REALITY target/xver` и fallback-контур через `nginx`: https://xtls.github.io/en/config/features/fallback.html
- [2026-04-10 22:48:10] Документация `wg-easy` по `AmneziaWG` подтверждает, что `AWG` добавляет транспортную обфускацию, требует совместимого host/module слоя и имеет параметры `Jc/Jmin/Jmax/S/H/I`, часть которых должна совпадать между сервером и клиентом: https://wg-easy.github.io/wg-easy/Pre-release/advanced/config/amnezia/
- [2026-04-10 23:00:59] Документация AmneziaWG 2.0 подтверждает дополнительные anti-DPI возможности поверх старого набора: опциональную маскировку первого байта под популярные UDP-протоколы через `H1-H4`, независимые `S3/S4`, обфускацию сессии через `I1-I5` и режим CPS. Это делает hardened `AWG` отдельным кандидатом, а не только "дефолтным WG в контейнере": https://docs.amnezia.org/documentation/amnezia-wg/
- [2026-04-10 23:00:59] Практические реализации `amnezia-wg-easy` подтверждают, что параметры `JC/JMIN/JMAX/S1/S2/H1-H4` могут передаваться как runtime configuration, но дефолтные значения и чужой image не считаются достаточными для нашего production hardening: https://github.com/w0rng/amnezia-wg-easy
- [2026-04-10 22:48:10] Исследование Censored Planet по TSPU фиксирует SNI/QUIC/IP-based blocking, packet drop и `RST/ACK`-поведение для отдельных сценариев. Для проекта это источник threat-model, а не доказательство универсальных численных порогов: https://censoredplanet.org/papers/tspu-imc22.pdf
- [2026-04-10 22:48:10] USENIX Security 2024 GFWeb по GFW показывает класс механизмов stateful HTTPS/SNI filtering, bidirectional injection и residual censorship. Для нашего проекта это полезная аналогия по механике `RST`/residual blocking, но не прямое доказательство поведения каждого российского оператора: https://www.usenix.org/system/files/usenixsecurity24-hoang.pdf

## Threat model для текущего этапа

- [2026-04-10 22:48:10] `SNI/DNS/IP mismatch`: домен в TLS/REALITY не должен выглядеть как чужой домен на неподходящем IP/ASN.
- [2026-04-10 22:48:10] `TLS/ALPN fingerprint`: опасны заметные транспортные признаки, например статичный `http/1.1` WebSocket-паттерн там, где он выглядит неестественно.
- [2026-04-10 22:48:10] `Burst/reconnect anomaly`: много коротких TLS-соединений, резкие повторные подключения и агрессивные socket-опции могут выглядеть хуже, чем один стабильный профиль.
- [2026-04-10 22:48:10] `Active probing`: узел не должен отвечать пустотой или странной ошибкой на неправильное рукопожатие; сканер должен видеть правдоподобный веб-сервис.
- [2026-04-10 22:48:10] `Long-running degradation`: короткий тест на 5-10 минут не закрывает риск деградации по времени суток, оператору, LTE/Wi-Fi и нагрузке.
- [2026-04-10 22:48:10] `IP/ASN blocking`: даже идеально настроенный transport не защищает от грубой блокировки IP/ASN, поэтому нужен fallback-план и резервный контур.
- [2026-04-10 22:48:10] `Admin exposure`: панель, подписки и будущие UI-модули не должны становиться лёгкими точками активного зондирования или компрометации.
- [2026-04-10 22:53:58] `Single-IP co-tenancy`: одновременная работа `Xray`-транспортов и дефолтного `AmneziaWG` на одном публичном IP может формировать общий поведенческий fingerprint. Если DPI классифицирует именно IP/flow-cluster, деградация может затрагивать все подключения к этому серверу, а не только один протокол.

## Технологии, которые используем в первую очередь

### 1. Правдоподобный edge и доменная модель

- [2026-04-10 22:48:10] Сохраняем два собственных домена: основной web/panel/sub домен и отдельный `REALITY`-домен.
- [2026-04-10 22:48:10] Оба домена должны резолвиться на ожидаемый VPS, без чужого публичного `SNI`.
- [2026-04-10 22:48:10] В `stealth`-контуре публичный `443` принадлежит `xray`, а `nginx` остаётся локальным fallback/web-слоем на `7443`.
- [2026-04-10 22:48:10] Fallback-сайт должен быть реальным и аккуратно обслуживаться, а не быть пустой заглушкой или ошибкой `502/404` на root.

### 2. `REALITY + Vision` как низколатентный stealth-профиль

- [2026-04-10 22:48:10] `stealth-xray` остаётся low-latency кандидатом: минимум внутренних moving parts, прямой `REALITY` ingress, стабильный `mobile-safe` fingerprint.
- [2026-04-10 22:48:10] Для `REALITY` используем только свои домены, стабильный fingerprint-профиль и controlled metadata tuning, а не случайный набор параметров в SQL.
- [2026-04-10 22:48:10] `Vision` не удаляется: его роль — быстрый профиль с низкой задержкой, который может быть лучше `XHTTP` там, где DPI не режет этот паттерн.

### 3. `REALITY + XHTTP` как более агрессивный anti-DPI профиль

- [2026-04-10 22:48:10] `stealth-xhttp` остаётся главным кандидатом для более агрессивных мобильных сетей.
- [2026-04-10 22:48:10] Используем `XHTTP` не как косметическую замену, а как другой behavioral profile: split up/down traffic, HTTP-compatible shape, header padding, `XMUX`, controlled buffering.
- [2026-04-10 22:48:10] Сначала закрепляем conservative `mobile-safe`, затем тестируем отдельные пресеты: `packet-up`, `stream-up`, `stream-one`, H2/H3 где клиент и сеть это поддерживают.
- [2026-04-10 22:48:10] Для скорости отдельно отслеживаем tradeoff: `XHTTP` может быть стабильнее по сигнатуре, но не обязан всегда быть быстрее `Vision`.

### 4. Controlled tuning вместо хаотичной рандомизации

- [2026-04-10 22:48:10] Все параметры transport tuning должны идти через metadata/preset слой, а не через ручную правку giant SQL.
- [2026-04-10 22:48:10] Базовые preset-имена: `default`, `mobile-safe`, `low-latency`, `aggressive-stealth`.
- [2026-04-10 22:48:10] Для каждого preset фиксируются влияние, риски и acceptance-критерии.
- [2026-04-10 22:48:10] Randomization допускается только внутри безопасных диапазонов и только если она не создаёт новый сильный fingerprint.

### 5. Мультипрофильность и failover

- [2026-04-10 22:48:10] Следующий целевой профиль после закрытия `C1` — `stealth-multi`: несколько проверенных transport-путей в одном install-контуре.
- [2026-04-10 22:48:10] Первый production-релиз не должен включать хаотично все возможные протоколы. Нужна маленькая матрица: основной профиль, fallback-профиль, резервный контур.
- [2026-04-10 22:48:10] `Trojan gRPC`, `WS TLS`, classic `XHTTP` и другие существующие подключения сохраняются как fallback/compatibility layer, но не считаются главным anti-DPI направлением без отдельной приёмки.
- [2026-04-10 22:48:10] Если один IP попадает под грубый IP/ASN блок, transport tuning не решит проблему полностью; для этого нужен резервный сервер или отдельный маршрут.
- [2026-04-10 22:53:58] Для production не смешиваем дефолтный `AWG` и основной `Xray`-контур на одном IP без отдельной co-tenancy приёмки. Безопасная стратегия: сначала изолировать `AWG` на резервный сервер или отдельный IP, затем возвращаться к совместному размещению только после hardening и тестов.
- [2026-04-10 23:00:59] Hardened `AWG` допускается как будущий кандидат даже на основном IP, но только если он не использует обычный дефолтный паттерн, имеет сгенерированные параметры, закрытый UI, state/backup/verify и проходит co-tenancy acceptance рядом с `Xray`.

## Защита от RST и деградации

### Что реально можно сделать

- [2026-04-10 22:48:10] Снижать вероятность триггера: свои домены, корректный `SNI`, правдоподобный TLS/HTTP shape, меньше burst/reconnect, стабильный fallback-сайт.
- [2026-04-10 22:48:10] Уменьшать эффект разрыва: несколько профилей, быстрый reconnect, понятный client handoff, long-running acceptance и profile-specific troubleshooting.
- [2026-04-10 22:48:10] На сервере использовать базовую TCP hygiene: `conntrack`/firewall для явно invalid-пакетов, аккуратные sysctl-настройки и сбор `tcpdump`/`ss`-артефактов при инцидентах.
- [2026-04-10 22:48:10] В `acceptance` сохранять признаки вокруг сбоя: время, профиль, оператор/тип сети, `ss -tinp`, `ss -s`, `journalctl`, клиентский лог.
- [2026-04-10 22:53:58] Для single-IP co-tenancy отдельно тестировать четыре режима: только `Xray`, только `AWG`, `Xray+AWG idle`, `Xray+AWG under load`. Если массовые `RST`/drop появляются только в смешанном режиме, это считается отдельным блокирующим риском совместного размещения.

### Что нельзя честно обещать

- [2026-04-10 22:48:10] Нельзя гарантировать, что серверный firewall отфильтрует все forged `RST`: если on-path middlebox видит sequence/window и отправляет валидный `RST` клиенту, клиентская сторона тоже может разорвать соединение.
- [2026-04-10 22:48:10] Нельзя считать `RST-drop` патч или blanket-block `RST` универсальным решением: это может ломать нормальное TCP-поведение и маскировать настоящую причину деградации.
- [2026-04-10 22:48:10] Нельзя закрыть `Block C1` только серверным `PASS`; нужна живая мобильная приёмка и reconnect-сценарии.

## Аутентификация и защита админского контура

- [2026-04-10 22:48:10] SSH: переходить к ключам, ограничивать password-login, сохранять доступный emergency-path только осознанно.
- [2026-04-10 22:48:10] Панель `3x-ui`: длинный случайный base path, сильные учётные данные, ротация после тестовых установок, отсутствие публикации секретов в docs.
- [2026-04-10 22:48:10] Web-sub и subscription endpoints: случайные пути, возможность ротации, проверка отсутствия старых upstream/user-facing утечек.
- [2026-04-10 22:48:10] Будущий `AWG UI`: по умолчанию не публиковать голый HTTP наружу; только localhost-bind, reverse proxy/TLS или явно включённый restricted mode.
- [2026-04-10 22:48:10] Следующий hardening-slice после transport C1: optional IP allowlist, fail2ban/rate-limit, audit логов входа, подготовка reverse-proxy auth layer для будущих UI-модулей.

## AmneziaWG в общей стратегии

- [2026-04-10 22:48:10] `AWG` не должен заменять `Xray`, а добавляется как отдельный optional transport module.
- [2026-04-10 22:48:10] Первый безопасный rollout `AWG` — на резервном сервере, чтобы не рисковать основным каналом связи.
- [2026-04-10 22:53:58] Причина такого rollout: дефолтный `AWG` на том же IP, где уже живут `Xray`-транспорты, может усиливать общий fingerprint сервера и провоцировать IP-level или residual деградацию.
- [2026-04-10 22:48:10] Параметры `Jc/Jmin/Jmax/S1-S4/H1-H4/I1-I5` должны генерироваться и сохраняться как state, а не быть одним вечным хардкодом.
- [2026-04-10 22:48:10] `AWG` требует отдельного verify: kernel module/tools, `/dev/net/tun`, `ip_forward`, NAT, firewall, UDP-доступность, client config, backup/restore.
- [2026-04-10 22:48:10] `AWG` не переводит весь проект в Docker-first. Для платформы сохраняется `bare-metal-first core`, а Docker используется как optional module там, где это действительно упрощает сопровождение.

## Hardened AWG как будущий кандидат

- [2026-04-10 23:00:59] `AWG` не списывается из-за риска co-tenancy. Списывается только дефолтный неуправляемый запуск без собственной state-модели, auth hardening и acceptance.
- [2026-04-10 23:00:59] Целевой `AWG`-модуль должен генерировать уникальные `Jc/Jmin/Jmax/S1-S4/H1-H4/I1-I5` в безопасных диапазонах, сохранять их в state и восстанавливать через backup/restore.
- [2026-04-10 23:00:59] Для `H1-H4` предпочтителен не статичный шаблон, а генерация/подбор под разрешённые значения и модель маскировки; один и тот же набор на все установки запрещён.
- [2026-04-10 23:00:59] `AWG` UI не должен публиковаться наружу как открытая web-панель: по умолчанию localhost-only или reverse proxy/TLS с отдельным auth/rate-limit.
- [2026-04-10 23:00:59] Перед совместным размещением с `Xray` обязательны четыре режима проверки: `Xray-only`, `AWG-only`, `Xray+AWG idle`, `Xray+AWG under load`.
- [2026-04-10 23:00:59] Если hardened `AWG` проходит эти проверки без усиления `RST`/drop-деградации, его можно рассматривать как часть `stealth-multi`/resilience платформы. Если нет — оставлять на отдельном IP/резервном сервере.

## Критерии готовности `Block C1`

- [2026-04-10 22:48:10] `stealth-xray` и `stealth-xhttp` проходят install, strict `verify`, `acceptance` и восстановление runtime context.
