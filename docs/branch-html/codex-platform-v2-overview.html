Что важно понимать

Назначение

Разработка модульной платформы с выбором профилей: classic, stealth-xray, stealth-xhttp, будущий stealth-multi и будущий hardened-awg.

Где эта ветка в общем процессе

Главная рабочая ветка нового платформенного слоя: anti-DPI hardening, профили, fallback и будущая модульность.

Правило: если ветка не помечена как stable или active implementation, её нельзя использовать как основной источник установки.

Технический профиль

REALITY/Vision для скорости, REALITY/XHTTP для снижения сигнатурности, подготовка нескольких проверенных fallback-путей.

Почему это важно

Мы разделяем stable, implementation, historical и upstream-ветки, чтобы не смешивать рабочую установку, эксперименты, архитектурные заметки и внешние источники.

Состояние ветки

Уже сделано

  • Добавлен selection-layer для выбора профилей без ломки classic-поведения.
  • Добавлен acceptance stage: установка должна пройти проверку готовности.
  • Зафиксированы mobile-safe tuning и отдельные профили stealth-xray / stealth-xhttp.
  • Заведены документы по DPI, RST/drop, single-IP co-tenancy и mobile whitelist fallback.
  • Сформирована стратегия: несколько управляемых маршрутов вместо одного “магического” протокола.

Риски и ограничения

  • Абсолютной невидимости не обещаем: цель - снижать сигнатурность и быстро переключаться при деградации.
  • AWG и Xray на одном IP могут давать корреляционный риск.
  • Mobile whitelist может блокировать IP/ASN, а не конкретный протокол.
  • Слишком агрессивная обфускация может ухудшить скорость.

Как проверять

  • Install на чистом staging-сервере.
  • Проверка x-ui, nginx, сертификатов, inbound JSON и service health.
  • Клиентские тесты V2RayN / v2rayNG: Wi-Fi, LTE, reconnect, смена сети.
  • Нагрузочный smoke: несколько 4K-потоков, web, мессенджеры, долгий тест 30-60 минут.
  • Сравнение логов: closed pipe, read/write, TLS/REALITY errors, nginx/x-ui restarts.

Что нельзя делать с этой веткой

  • Не сливать platform-v2 в main без regression.
  • Не добавлять протоколы без проверяемого выигрыша.
  • Не хранить в GitHub личные заметки, пароли, реальные тестовые домены.
  • Не включать AWG по умолчанию до hardening-реализации.
