# PROJECT_CONNECTIVITY_REQUIREMENTS

## Назначение

- [2026-04-10 20:07:20] Этот документ фиксирует проектные требования к устойчивой связи и рабочему доступу к ресурсам в условиях DPI-блокировок.
- [2026-04-10 20:07:20] Документ является обязательным продуктовым контекстом для текущего этапа `Xray/DPI` и следующей интеграции `AWG`.

## Непереговорные цели

- [2026-04-10 20:07:20] Главная цель форка: стабильная связь для критичных коммуникационных сервисов и работы даже при деградации отдельных протоколов.
- [2026-04-10 20:07:20] Мессенджеры и базовый web-доступ должны оставаться доступными в повседневном сценарии.
- [2026-04-10 20:07:20] Система должна быть управляемой и предсказуемой: любой сбой должен быстро диагностироваться и обходиться fallback-маршрутом.

## Ограничения

- [2026-04-10 20:07:20] Инфраструктура ограничена бюджетом: целевая модель — небольшое число серверов, а не массовое масштабирование по множеству IP.
- [2026-04-10 20:07:20] Блокировки по IP/ASN возможны независимо от качества конфигурации; это учитывается в архитектуре как внешний риск.
- [2026-04-10 20:07:20] Абсолютно «неблокируемого» transport-профиля не существует; целевой результат — максимальная живучесть и быстрое переключение.
- [2026-04-10 22:53:58] Размещение нескольких разных anti-DPI/VPN стеков на одном IP считается отдельным риском: даже если каждый стек работает отдельно, их совместный fingerprint может ухудшать поведение у DPI.
- [2026-04-10 23:05:05] Мобильный whitelist/IP-level drop считается отдельным внешним ограничением: если оператор не пропускает прямой VPS IP, один только transport hardening не гарантирует связь.

## Ключевые архитектурные выводы

- [2026-04-10 20:07:20] Транспортная устойчивость важнее одного «идеального» протокола: форк должен развиваться как multi-transport платформа.
- [2026-04-10 20:07:20] Основной контур (`Server A`) должен поднимать несколько каналов одновременно: `stealth-xhttp`, `trojan-grpc`, `ws-tls`, `reality`.
- [2026-04-10 20:07:20] Резервный контур (`Server B`) должен служить failover-площадкой и не зависеть от состояния основного IP.
- [2026-04-10 23:05:05] Для мобильных сетей нужен отдельный fallback-класс поверх обычного `Server A/Server B`: профиль или маршрут, который проверяется именно против whitelist/IP-level drop, а не только против DPI-сигнатур.
- [2026-04-10 20:07:20] `AWG` вводится сначала только на резервном сервере, чтобы не рисковать основным каналом связи.
- [2026-04-10 22:53:58] Дефолтный `AWG` не должен считаться безопасным соседом для основного `Xray` на одном IP до отдельной проверки `single-IP co-tenancy`.
- [2026-04-10 23:00:59] Hardened `AWG` остаётся допустимой целью для основного IP в будущем, если он имеет уникальные параметры обфускации, закрытый admin UI и проходит co-tenancy acceptance рядом с `Xray`.
- [2026-04-10 20:07:20] Для снижения сигнатурности и RST-чувствительности применяются controlled preset-подходы (`mobile-safe`) и консервативные runtime-параметры вместо хаотического random-tuning.
- [2026-04-10 20:07:20] Защита хоста и панели обязательна (SSH keys, ограничение доступа, ротация учётных данных), но сама по себе не заменяет anti-DPI стратегию.

## Защита от RST и деградации

- [2026-04-10 20:07:20] Текущая практическая стратегия — уменьшать burst/reconnect-поведение и повышать устойчивость длинных сессий.
- [2026-04-10 20:07:20] Профили должны тестироваться на `Wi-Fi` и `LTE` отдельно, включая переключения сети и многократные reconnect.
- [2026-04-10 20:07:20] Любой transport, который работает только в коротком тесте, не считается принятым до long-running приёмки.
- [2026-04-10 22:53:58] Если при одновременной нагрузке `Xray` и `AWG` на одном IP возникают массовые `RST`/drop-симптомы, это трактуется как co-tenancy incident, а не как баг только одного протокола.

- [2026-04-10 20:07:20] Для каждого полевого сбоя сохранять: время (МСК), сеть (LTE/Wi-Fi), активный профиль, симптом, клиентский лог вокруг сбоя.
- [2026-04-10 22:53:58] Для co-tenancy incident дополнительно сохранять, какие сервисы на IP были активны одновременно: `Xray`, `AWG`, web/panel, нагрузка, idle/load состояние.
