# STAGING_RESET

## Назначение

- [2026-04-05 07:35:53] `stage=reset` нужен для повторяемого возврата staging-ноды в чистое состояние перед новым полным install-прогоном.
- [2026-04-05 07:35:53] Этот режим не заменяет обычный `uninstall`, а дополняет его staging-специфичной обвязкой: pre-reset артефактами, защитным подтверждением и post-check после очистки.
- [2026-04-05 07:35:53] Режим рассчитан именно на тестовый VPS в цикле `Windows -> GitHub -> Linux staging`.

## Команды

- [2026-04-05 07:35:53] Безопасный preview без изменения системы: `sudo bash ./x-ui-pro-updated.sh -stage reset -dry_run yes -debug yes -keep_artifacts yes`
- [2026-04-05 07:35:53] Реальный reset: `sudo bash ./x-ui-pro-updated.sh -stage reset -confirm_reset yes -debug yes -keep_artifacts yes`
- [2026-04-05 07:35:53] Без `-confirm_reset yes` destructive reset не запускается и завершается защитной ошибкой.

## Что делает reset

- [2026-04-05 07:35:53] Считывает текущий runtime-контекст установленной ноды, если он ещё доступен через `x-ui.db` и `nginx`.
- [2026-04-05 07:35:53] Сохраняет pre-reset артефакты: `nginx.conf`, `includes.conf`, `stream.conf`, `x-ui.db`, web-sub файлы и состояние TCP-listener'ов.
- [2026-04-05 07:35:53] Останавливает `nginx`, `x-ui` и процесс `sub2sing-box`.
- [2026-04-05 07:35:53] Выполняет полный `uninstall` текущего install-контура.
- [2026-04-05 07:35:53] Дополнительно удаляет остаточные каталоги `/etc/x-ui`, `/usr/local/x-ui`, `/etc/nginx`, `/var/www/subpage`, `/etc/letsencrypt`, `/var/lib/letsencrypt`, `/var/log/letsencrypt`, а также бинарники `x-ui` и `sub2sing-box`.
- [2026-04-05 07:35:53] Выполняет post-check: сервисы, бинарники, пути и порты `80/443` должны оказаться в чистом состоянии.

## Что reset сознательно не делает

- [2026-04-05 07:35:53] Не трогает произвольные пользовательские файлы вне install-контура форка.
- [2026-04-05 07:35:53] Не используется как production-процедура сопровождения.
- [2026-04-05 07:35:53] Не заменяет отдельный regression-аудит `backup.sh`, `restore` и стандартного `-uninstall yes`.
- [2026-04-05 07:35:53] Не сбрасывает `UFW` в дефолт и не выполняет сетевую очистку, которая может рискованно повлиять на SSH-доступ.

## Правильный порядок использования

- [2026-04-05 07:35:53] Сначала завершить локальный этап на Windows и отправить изменения в GitHub.
- [2026-04-05 07:35:53] Затем на staging выполнить preview reset через `-dry_run yes` и убедиться, что сценарий соответствует ожиданиям.
- [2026-04-05 07:35:53] После этого выполнить реальный `stage=reset -confirm_reset yes`.
- [2026-04-05 07:35:53] Только затем запускать новый полный install-сценарий и последующие regression-проверки.
